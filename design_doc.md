# 周末规划助手 Agent 设计文档

## 1. 产品定位

**一句话描述：** 用户用自然语言描述周末出行目标，助手自动理解人群、时间、预算、距离、天气和餐饮约束，生成可确认、可调整、可执行的周末活动方案。

**当前定位：** 这是一个“Agent 化规划助手”，不是全自动周末管家。查询、筛选、排序、路线和天气判断可由助手自动完成；预订、下单、支付、分享等关键动作必须由用户确认。

**核心场景：**

> “带 5 岁孩子出来放电。帮我排一个今天下午的行程，需要一个能消耗孩子体力的室内或户外场馆，周边步行距离内要有适合全家聚餐的高分餐厅，大人能顺便坐着休息喝咖啡。”

系统会识别为亲子高能半日场景；如果当前是工作日，会自动将“今天下午”在周末规划语境下转为最近周六下午，并用对应日期查询天气预报。

## 2. Demo 当前能力

| 模块 | 当前能力 |
|---|---|
| 意图解析 | 识别人群、儿童年龄、人数、预算、交通、时间段、强度、宠物、换方案诉求 |
| 时间推断 | 默认从当前时间之后规划；工作日的周末场景自动指向最近周末下午 |
| POI 搜索 | 通过高德 POI 搜索活动、餐厅、景点、酒店，并支持详情补充 |
| POI 过滤 | 过滤医院、药房、政府、银行、加油站等不适合作为活动的 POI |
| 天气判断 | 使用 Open-Meteo 查询实时/指定日期/周末天气，并与行程日期对齐 |
| 推荐排序 | 综合距离、评分、价格、人群适配、天气和预算约束排序 |
| 方案生成 | 输出活动、餐厅、可选景点/酒店、路线、时间轴和人均预估 |
| 多轮调整 | 支持换餐厅、换活动、换近一点、预算低一些、换套方案 |
| 局部重绘 | “不想吃这家，换个餐厅”只替换餐厅，不重新完整思考 |
| 风险控制 | 预订/支付由用户确认；营业时间、闭馆、预约等信息做显式处理 |
| RAG 补充 | 提供北京周末路线静态知识库，可作为攻略参考源 |

## 3. Agent 流程

```text
用户输入自然语言需求
  ↓
意图解析：人群 / 时间 / 预算 / 距离 / 偏好 / 反馈类型
  ↓
上下文补齐：定位 / 当前日期 / 周末推断 / 上一轮方案状态
  ↓
并行工具调用：POI 搜索 / 天气查询 / POI 详情 / 路线估算
  ↓
候选过滤：无效 POI / 营业时间 / 儿童友好 / 距离 / 预算
  ↓
方案排序：距离优先、评分辅助、价格和可执行性约束
  ↓
生成方案卡：时间轴 + 活动 + 餐厅 + 路线 + 人均预估
  ↓
用户确认或反馈
  ↓
局部重绘 / 换套方案 / 预订确认 / 分享
```

### 3.1 完整规划

适用于首次输入、换套方案、宠物/两天一夜等明显改变需求的情况。完整流程会重新解析意图、调用工具、筛选候选并生成新方案。

### 3.2 局部重绘

适用于用户只否定局部节点的情况：

| 用户反馈 | 行为 |
|---|---|
| 不想吃这家 / 换个餐厅 | 保留活动和时间，只替换餐厅 |
| 不想去这个活动 / 换个活动 | 保留人群和餐饮偏好，替换活动 |
| 换个近一点的 | 保留原需求，收紧距离阈值并重排 |
| 预算低一些 | 保留原需求，优先低价活动和餐厅 |

当前已实现餐厅局部重绘：不会展示完整思考链，也不会重新生成整套规划，只在计划卡中标注“局部重绘：已更换餐厅”。

## 4. 关键策略

### 4.1 周末时间推断

| 场景 | 处理 |
|---|---|
| 工作日 + 周末/亲子/孩子/放电/约会等场景 | 默认规划最近周六下午 |
| 用户明确说“必须今天/现在/马上” | 尊重今天，不自动挪到周末 |
| 周末当天半日请求 | 默认规划当天 |
| 全天/两天一夜请求 | 查询周六/周日天气并输出跨天方案 |

### 4.2 天气对齐

天气查询不再固定用当前实时天气，而是跟随 `intent.targetDate`：

- 今天方案：查实时天气。
- 周六下午方案：查周六预报。
- 两天一夜/全天周末方案：查周六和周日天气。

### 4.3 距离收紧规则

“换个近一点的”会根据当前活动距离收紧：

| 当前活动距离 | 新阈值 |
|---|---:|
| > 8km | 5km 内 |
| 3–8km | 3km 内 |
| ≤ 3km | 2km 内 |

近距离模式下按距离优先排序，避免高评分远距离 POI 被顶上来。

### 4.4 特殊场馆规则

故宫等特殊场馆需要额外规则：

- 标注“需提前预约”。
- 标注“周一闭馆”。
- 使用更接近真实的开放时间和停止入院时间。
- 晚间或临近闭馆时不作为临时活动推荐。

## 5. 工具调用清单

| 工具/函数 | 作用 | 自动化权限 |
|---|---|---|
| `parse_intent` | 自然语言转结构化槽位 | 自动 |
| `fetchLocation` | 获取用户城市和坐标 | 需授权/可降级 |
| `searchAmapPOI` | 搜索活动、餐厅、酒店、景点 | 自动 |
| `fetchAmapPoiDetail` | 获取 POI 详情和营业信息 | 自动 |
| `fetchWeather` | 查询实时或指定日期天气 | 自动 |
| `filterValidActivities` | 过滤医院等无效活动 POI | 自动 |
| `filterOpenForStart` | 按营业时间过滤 | 自动 |
| `selectWithLLM` | 在候选中做最终选择 | 自动/可降级 |
| `renderPlanCard` | 生成计划卡 | 自动 |
| `localRepaintRestaurant` | 只替换餐厅节点 | 自动 |
| `executeBookings` | 执行预订/出票模拟 | 用户确认后 |
| `sharePlan` | 生成分享卡 | 用户确认后 |

## 6. 推荐质量约束

| 维度 | 规则 |
|---|---|
| 人群 | 亲子优先儿童乐园、亲子运动、科技馆、室内游乐场、儿童友好餐厅 |
| 距离 | 半日方案距离优先，默认优先 8km 内，换近一点时更严格 |
| 时间 | 活动、交通、餐饮必须能放进用户时间窗 |
| 预算 | 有预算时按活动+餐饮人均估算，不显示“待确认” |
| 天气 | 下雨/恶劣天气优先室内或有遮蔽场所 |
| 营业 | 闭馆、停止入院、休息中场馆不能临近安排 |
| 可调整 | 用户反馈后尽量局部修改，避免整套方案重跑 |

## 7. 用户界面原则

当前用户视角只保留对决策有帮助的信息：

- 方案标题、人群、预算、人均预估。
- 时间轴、活动、餐厅、路线距离和预计耗时。
- 继续问入口：换餐厅、换活动、换近一点、预算低一些、突发带狗、换套方案。
- 确认方案与跳转入口。

不在用户视角展示：

- 内部指标面板。
- 工具可靠性面板。
- 风险控制解释长文。
- 过多权限说明。
- 重复反馈按钮。

技术视角仍可保留 ReAct、工具调用和推理过程，用于演示 Agent 机制。

## 8. 评估指标

| 指标 | 说明 |
|---|---|
| 计划完成率 | 是否生成活动 + 餐厅 + 路线 + 时间轴 |
| 首轮采纳率 | 用户是否直接确认首个方案 |
| 局部重绘成功率 | 用户换餐厅/换活动后是否满意 |
| 硬约束违规率 | 是否出现医院、闭馆、过远、不适合儿童等问题 |
| 工具成功率 | POI、天气、详情接口是否成功返回 |
| 工具恢复率 | 工具失败后是否能降级生成方案 |
| 预算匹配度 | 人均预估是否符合预算 |
| 距离匹配度 | 是否满足“近一点/别太远”等约束 |
| 天气匹配度 | 天气日期是否与行程日期一致 |

## 9. 当前边界

- Demo 仍是前端单文件为主，适合演示 Agent 产品逻辑，不等同于生产系统。
- 高德 POI 详情中的营业时间字段可能不完整，关键场馆需要特殊规则兜底。
- 预订/支付为模拟流程，真实交易必须接入正式订单系统和用户授权。
- RAG 模块是静态路线知识库补充，不替代实时 POI 和实时营业状态。
