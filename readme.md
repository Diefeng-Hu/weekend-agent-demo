# 周末规划助手 Demo

这是一个面向周末出行决策的 Agent 化规划 Demo。用户输入一句自然语言需求，系统会自动理解人群、时间、预算、距离、天气、餐饮和路线约束，生成一张可确认、可调整的周末计划卡。

## 1. 项目定位

周末活动规划不是简单问答，而是一个多步骤任务：

```text
想法 → 意图理解 → 搜活动 → 搜餐厅 → 查天气 → 算路线 → 排时间 → 估预算 → 用户确认 → 预订/分享
```

本项目的目标不是“替用户全自动下单”，而是做一个可控的规划助手：

- 查询、筛选、排序和计划生成由助手自动完成。
- 预订、支付、分享等关键动作必须由用户确认。
- 用户不满意时优先局部调整，不轻易推翻整套方案。

## 2. 当前核心能力

| 能力 | 说明 |
|---|---|
| 自然语言意图解析 | 识别人群、孩子年龄、预算、时间、强度、交通、宠物、反馈类型 |
| 工作日转周末 | 工作日收到周末/亲子/放电等场景时，默认规划最近周六下午 |
| 天气日期对齐 | 周末方案查对应周末预报，不再误用当前实时天气 |
| 高德 POI 搜索 | 搜活动、餐厅、景点、酒店，并尝试获取 POI 详情 |
| 推荐质量过滤 | 过滤医院、药房、政府、银行等不适合作为活动的 POI |
| 营业时间处理 | 支持营业时间过滤，故宫等特殊场馆有预约/闭馆规则 |
| 人均费用预估 | 不显示“待确认”，按活动、餐饮、酒店等估算费用 |
| 多轮上下文继承 | 换近一点、预算低一些、换餐厅时保留原人群和时间 |
| 局部重绘 | “不想吃这家，换个餐厅”只换餐厅，不重新完整思考 |
| RAG 补充 | `rag_module.py` 提供北京周末路线静态知识库检索 |

## 4. Demo 体验路径

1. 打开 `demo.html`。
2. 选择快捷场景或直接输入需求，例如：

```text
带（5岁）孩子出来放电。帮我排一个今天下午的行程，需要一个能消耗孩子体力的室内或户外场馆，周边步行距离内要有适合全家聚餐的高分餐厅。路线要顺，大人能顺便坐着休息喝咖啡。
```

3. 查看生成的计划卡：活动、餐厅、路线、时间轴、人均费用。
4. 使用“继续问”调整：
   - 换一个餐厅
   - 换个活动
   - 换个近一点的
   - 预算低一些
   - 突发带狗
   - 换套方案
5. 点击确认方案后进入预订/计划卡流程。

## 5. 关键交互规则

### 5.1 周末时间推断

| 输入语境 | 行为 |
|---|---|
| 工作日 + 亲子/周末/放电/约会/聚会 | 默认规划最近周六下午 |
| 明确“必须今天/马上/现在” | 按今天当前时间规划 |
| 周末当天半日请求 | 按当天规划 |
| 两天一夜 | 生成跨天方案并加入住宿 |

### 5.2 换近一点

| 当前活动距离 | 新距离阈值 |
|---|---:|
| > 8km | 5km 内 |
| 3–8km | 3km 内 |
| ≤ 3km | 2km 内 |

### 5.3 换餐厅局部重绘

用户输入：

```text
不想吃这家，帮我换个餐厅
```

系统行为：

- 不重新展示完整思考链。
- 保留活动、日期、时间、人群和天气。
- 只从已有候选餐厅中替换餐厅。
- 新计划卡标注“局部重绘：已更换餐厅”。

## 6. 推荐质量约束

- 亲子场景优先儿童乐园、亲子运动、科技馆、室内游乐场、儿童友好餐厅。
- 活动 POI 过滤医院、医疗、药房、政府、银行、加油站、维修等无关地点。
- 故宫等特殊场馆标注预约、闭馆和停止入院时间，不安排临近闭馆时段。
- 半日计划距离优先，避免为了评分推荐远距离 POI。
- 天气与行程日期对齐，下雨时优先室内或有遮蔽场景。

## 7. 技术说明

- 前端主逻辑集中在 `demo.html`，便于演示和快速迭代。
- POI 数据优先使用高德搜索和详情接口，失败时可降级到 Mock 数据。
- 天气使用 Open-Meteo。
- LLM 可选；没有 Key 时使用规则解析和本地排序策略。
- RAG 是路线知识库补充，适合攻略型问答或北京路线推荐，不替代实时 POI。

## 8. 当前边界

- 这是演示级 Demo，不是生产级交易系统。
- 预订/支付为模拟或跳转入口，真实交易必须接入用户授权和订单系统。
- 部分 POI 营业时间字段依赖第三方接口返回，特殊场馆需要规则兜底。
- 用户视角已隐藏大部分技术面板，技术视角仍用于解释 Agent 工作机制。
